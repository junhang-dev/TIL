# [TIL] EC2 인스턴스 생명주기: '중지'와 '종료'의 차이 (비용 최적화)

**Date:** 2025-11-07

## 1. 학습 목표

EC2 인스턴스 사용 시 비용이 과금되는 원리를 이해하고, '중지'와 '종료'의 명확한 차이를 파악하여 개발 서버 비용을 최적화한다.

## 2. '중지(Stop)' vs '종료(Terminate)'

이 둘은 완전히 다른 개념이며, 잘못 사용하면 비용 폭탄을 맞거나 데이터를 영구적으로 잃을 수 있다.

### 1) 인스턴스 중지 (Stop)

* **개념:** 컴퓨터의 '전원 끄기'와 같다.
* **특징:**
    * **컴퓨팅 비용 미부과:** 인스턴스가 꺼져있는 동안 `t3.small` 같은 vCPU, Memory 사용료가 과금되지 **않는다.**
    * **데이터 보존:** 연결된 EBS 볼륨(하드디스크)은 그대로 보존된다. 즉, 내가 설치한 Ubuntu, Docker, Firecrawl 소스 코드 모두 안전하게 남아있다. (단, EBS 스토리지 비용은 소액 부과됨)
    * **IP 회수:** **퍼블릭 IPv4 주소를 회수당한다.** (이것이 핵심)
* **사용 시점:** 개발이 끝나고 잠시 서버를 끌 때. "나중에 다시 이어서 작업해야지" 할 때 사용.

### 2) 인스턴스 종료 (Terminate)

* **개념:** 컴퓨터를 **'버리는 것'** (폐기)과 같다.
* **특징:**
    * **영구 삭제:** 인스턴스와 연결된 EBS 볼륨이 **모두 영구적으로 삭제된다.**
    * **복구 불가능:** 삭제된 후에는 절대로 되돌릴 수 없다.
    * **비용 중단:** 당연히 모든 비용 과금이 중단된다.
* **사용 시점:** 프로젝트가 완전히 끝나서 이 서버가 더 이상 필요 없을 때.

## 3. 나의 워크플로우 (비용 최적화)

1. **개발 종료 시:**
    * EC2 터미널에서 `docker compose down` 실행.
    * AWS 콘솔에서 `Firecrawl_server` 인스턴스 선택.
    * **[인스턴스 상태] > [인스턴스 중지]** 클릭.
2. **개발 재개 시:**
    * AWS 콘솔에서 **[인스턴스 시작]** 클릭.
